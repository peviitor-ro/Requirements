ID: EPIC-013 Epic
PERFORMANȚĂ MAXIMĂ
Viteză maximă la orice scară. peviitor.ro livrează rezultale de căutare în milisecunde, indiferent de volumul de date. Performanța este o cerință, nu un moft.

Hardware actual

Dispozitiv RAM Rol
Raspberry Pi 5 16 GB SOLR — motorul de căutare
Raspberry Pi 5 4 GB API — serverul de backend
Raspberry Pi 4 2 GB Test — mediu izolat de testare
Raspberry Pi 400 4 GB Backup — preia orice rol în caz de cădere

Platforma trebuie să suporte toate companiile din România și toate anunțurile de angajare — sute de mii de job-uri active simultan, căutări concurente și indexare continuă.

Principiu

Performanța ridicată și resursele minime nu sunt termeni opuși — sunt două fațete ale aceleiași abordări. Un sistem bine scris, optimizat și fără balast va fi întotdeauna rapid, indiferent de hardware-ul pe care rulează.

Fiecare componentă este aleasă sau construită pentru eficiență maximă: SOLR indexează și răspunde în milisecunde, front-end-ul este static și ușor, API-ul este minimalist.

Metrici de performanță

Operațiune Țintă Condiții
Căutare full-text < 50 ms 5 M+ documente, orice cluster node
Filtrare + căutare < 100 ms orice combinație de filtre
Indexare job nou < 10 ms per document
Încărcare pagină web < 500 ms Lighthouse, orice viteză de rețea
API response < 100 ms inclusiv query SOLR
Autocomplete sugestie < 20 ms după primul caracter

Optimizări SOLR

  • Heap dimensionat corect — nu se alocă mai mult de 50% din RAM pentru heap-ul SOLR; restul rămâne pentru cache-ul sistemului de operare
  • Cache-uri — filterCache, queryResultCache, documentCache configurate pentru hit-rate > 90%
  • Schema restrânsă — numai câmpurile necesare sunt indexate; câmpurile doar stocate (lastScraped, scraperFile) nu încetinesc indexarea
  • Index segment merge — policy care menține numărul de segmente sub control fără a consuma CPU în momentele de vârf
  • Replication factor 1 — în regim minim, o singură copie a indexului este suficientă; replica se adaugă doar când traficul o cere

Optimizări front-end

  • Static site — front-end-ul este livrat ca fișiere statice (HTML, CSS, JS), fără server-side rendering
  • Fără framework greu — JS vanilla sau lightweight, fără React/Vue (decât dacă este strict necesar)
  • Assets optimizate — CSS minified, JS bundled, imagini comprimate WebP, fonturi subsetate
  • CDN — fișierele statice sunt servite prin GitHub Pages CDN, cu cache pe termen lung
  • Lazy loading — rezultatele sunt încărcate progresiv: primele 20 de rezultate apar imediat, restul la cerere

Optimizări API

  • Răspunsuri JSON minimale — se returnează doar câmpurile cerute, fără împachetări inutile
  • Paginare — default 20 de rezultate per request, max 100
  • Caching HTTP — headere Cache-Control și ETag pentru răspunsuri care nu se schimbă des
  • Query-uri eficiente — se folosește SOLR Query API direct, fără strat intermediar care adaugă latență

Fără balast

Performanța vine și din ceea ce nu facem:

  • Fără reclame — niciun script de advertising care încetinește pagina
  • Fără trackere — niciun Google Analytics, Facebook Pixel sau alt script terț
  • Fără fonturi externe grele — fonturile sunt subsetate la caracterele necesare (latină, diacritice)
  • Fără dependințe inutile — fiecare bibliotecă adăugată este justificată și cântărită

Testare de performanță

  • Benchmark SOLR — test periodic cu seturi de date standardizate pentru măsurarea timpilor de căutare și indexare
  • Load test — simulare trafic real: zeci de căutări simultane, filtre, autocomplete
  • Lighthouse CI — integrat în GitHub Actions pentru a monitoriza performanța front-end-ului
  • Regresie — orice PR care afectează performanța este semnalat automat

Cluster actual

Cele 4 Raspberry Pi funcționează ca un cluster SOLR. Configurarea permite:

  • High availability — dacă nodul principal cade, secundarul preia automat
  • Testare izolată — Pi 4 de 2 GB rulează mediul de test, complet separat de producție
  • Backup instant — Pi 400 backup poate prelua în câteva minute
  • Replicare — indexul SOLR este replicat pe minim 2 noduri

Scalare viitoare

Când traficul sau volumul de date va depăși capacitatea actuală, scalarea se face treptat:

  • Pasul 1 — SSD extern NVMe pe fiecare Pi (de la SD card la NVMe)
  • Pasul 2 — al 5-lea Pi (replică suplimentară pentru shard)
  • Pasul 3 — SOLR Cloud cu sharding pe mai multe noduri
  • Pasul 4 — migrare pe echipamente dedicate (dacă și când va fi necesar)

Fiecare pas este opțional și se face doar când metricile arată că este necesar.
